## Contribution Process

### Reporting Bugs

1. **Check existing issues**: Search for similar reports
2. **Create detailed issue**:
   - Clear title
   - Steps to reproduce
   - Expected vs actual behavior
   - Environment details (ARK version, K8s version)
   - Relevant logs

### Suggesting Features

1. **Check existing requests**: Search for similar ideas
2. **Create feature request**:
   - Clear description
   - Use case / problem it solves
   - Proposed solution
   - Alternatives considered

### Submitting Code Changes

1. **Fork the repository**
2. **Create a branch**: `git checkout -b feature/my-feature`
3. **Make changes**: Write code, add tests, update docs
4. **Test your changes**: Ensure all tests pass
5. **Commit**: Use [conventional commits](https://www.conventionalcommits.org/)
   ```
   feat: add new feature
   fix: resolve bug in X
   docs: update documentation
   ```
6. **Push**: `git push origin feature/my-feature`
7. **Create Pull Request**: Fill out the PR template

### Pull Request Guidelines

- **Clear title**: Describe what the PR does
- **Description**: Explain why and how
- **Link issues**: Reference related issues
- **Tests**: Include tests for new features
- **Documentation**: Update docs if needed
- **Keep focused**: One logical change per PR

## Code Style

### Go Code
- Follow `gofmt` formatting
- Run `go vet` and `golangci-lint`
- Follow existing code patterns

### Python Code
- Follow PEP 8
- Use type hints where appropriate
- Run linters and formatters

### TypeScript/JavaScript
- Follow project ESLint rules
- Use TypeScript for new code
- Match existing code style

### Documentation
- Use clear, concise language
- Include examples where helpful
- Link related content
- Follow Diataxis principles (explanations, tutorials, how-to, reference)

## Commit Message Format

ARK uses [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**Types**:
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Code style (formatting, etc.)
- `refactor`: Code refactoring
- `test`: Adding or updating tests
- `chore`: Maintenance tasks

**Examples**:
```
feat(agents): add support for custom parameters
fix(api): resolve timeout issue in queries
docs(tutorials): add multi-agent team tutorial
```

## Testing

- **Run existing tests**: `make test` (if available)
- **Add tests**: Include tests for new features
- **E2E tests**: Use Chainsaw for E2E testing (see [Testing Guide](/developer-guide/testing))

## Documentation Contributions

We especially welcome documentation improvements! See [Issue #338](https://github.com/mckinsey/agents-at-scale-ark/issues/338) for areas needing work.

### Documentation Style

- **Be clear and concise**: Write for your audience
- **Include examples**: Show, don't just tell
- **Link related content**: Connect concepts
- **Use proper formatting**: Code blocks, lists, etc.

### Documentation Types

- **Explanations**: Conceptual understanding (why/when)
- **Tutorials**: Step-by-step learning (how to learn)
- **How-to Guides**: Problem-solving (how to accomplish)
- **Reference**: Technical specifications (what/how)

See the [Explanations section](/explanations) for examples of each type.
